class AndPredicate implements IRunnerPredicate {

These kinds of function objects that are constructed with additional parameters are known as parameterized function objects, and they are both very useful and very common.

// Interface for two-argument function-objects with signature [A1, A2 -> R]
interface IFunc2<A1, A2, R> {
  R apply(A1 arg1, A2 arg2);
}

// In IList<T> => args are a function(T,U => U), base value 
<U> U foldr(IFunc2<T, U, U> func, U base);
 
// In MtList<T>
public <U> U foldr(IFunc2<T, U, U> func, U base) {
  return base;
}
 
// In ConsList<T>
public <U> U foldr(IFunc2<T, U, U> func, U base) {
  return func.apply(this.first,
                    this.rest.foldr(func, base));
}

Important Note on Java Function Module

The IFunc and IFunc2 interfaces are very useful, to the point that recent versions of Java have predefined versions of them for us.

  • IPred<T> => Is actually Predicate<T>,
  • IFunc<A, R> => Is actually Function<A, R>,
  • IFunc2<A1, A2, R> => Is Actually BiFunction<A1, A2, R>
    As defined in the java.util.function package.

Visitors

Abstractions so far:

Over the last few lectures, we have seen two powerful abstraction mechanisms:

  • function objects allow us to abstract over behavior, giving us the flexibility of higher-order functions in an object-oriented setting,
  • generics allow us abstract over types, defining once-and-for-all families of related types like IList.
  • We can even combine the two, defining interfaces like IFunc<A, R> that describe all function objects that take an argument of type A and return a result of type R.
  • We saw examples of how to define function objects over simple data: for instance, IFunc<Book, String> is the interface for a function object whose apply method takes a Book and returns a String.
  • We could then take those function objects and map() them across lists. So far, so good.

How about for complex data types? In particular, what happens if we try to write a function object that takes a value of a union data type?
