Function Subtyping

Subtyping for functions is counterintuitive.

When is a function a subtype of another?

  • Important in higher order function languages.
  • If function expects t1->t2 can you pass t3->t4 instead?

e.g.

flip has the type disMoved expects for f.
Can pass in record with extra fields of p.

BUT...
now pass in flipGreen. Flips x and y, adds color field green.

Return type of flipGreen is {x:real, y:real, color:string}. but distMoved expected {x:real,y:real}
Nothing goes wrong.

  • If ta <: tb ,then t->ta <: t->:tb
    A function can return 'more than it needs to'
    Jargon return types are 'covariant'.

Argument type though: Does not work!
flipIfGreen is expecting x:real, y:real, color: string => but called with a color.,

Unsound! => ta<:tb does not allow ta->t <: tb->t
Thus cannot allow subtyping of argument types.

But the other way works: allow subtype on function argument types if backwards
If tb<:ta, then ta->t <: tb->t
A function can assume 'less than it needs to' about arguments
Jargon: argument types are contravariant
Pass in super type object

Can do both:
flipXMakeGreen => consume x, return x,y,color.
Can pass this function into a higher order function requiring: (x:real, y:real) -> (x:real, y:real)
If t3<:t1 AND t2<:t4 => then t1->t2 <: t3->t4

General rule:
If t3<:t1 AND t2<:t4 => then t1->t2 <: t3->t4
Where t3 and t1 are argument types, t2 t4 are
Thus 'Function subtyping is contravariant in arguments and covariant in results'

Subtyping for OOP

Now apply to Java/C#.
Class Names are also types => Subclasses are subtypes.

Class names are types.
Subclasses are subtypes
Substitution principle: instance of subclass should be usable in place of instance superclass

Objects are records holding field and methods.
Fields mutable, methods immutable functions that can access self.
=> Can have subtyping on methods (like functions, rules for subtyping functions here apply)

COULD design type system using types like record types.

  • Subtypes could have extra fields and methods.
  • Overriding methods could have contravariant arguments and covariant results compared to method overriden
    • "Can pass in args with fewer than the superclass, but return more types than the superclass"
    • sound only because method slots are immutable

Actual Java and C#:

  1. Types are class names and subtyping are explicit subclasses.
  2. Subclass can add field and methods.
  3. Subclass can override a method with a covariant return type
    1. NO contravariant arguments, instead makes it a non-overriding method of the same name.
      Instead they consider you can make another method with same name with different inputs. (non-overriding method of same name)

Classes vs Types

Classes are about behavior.
Types are about interfaces (what you can take and return).

A class defines an objects' behavior. Subclassing inherits behavior and changes via extension and overriding.

A type describes an objects' methods' argument/result types. A subtype is substitutable in terms of its field/method types.

Java/C# confuse them by requiring subclasses to be subtypes.
Class name is both a class and type.
This confusion is convenient in practice.

optional topic 1:
Java and C# are sound - do not allow subtypes to do things leading to method missing or accessing field at wrong type.
e.g.

  • Subclass declare a field name already from superclass.
  • 2 classes can use any two types for field name
  • Instance of subclass have 2 fields with same name
  • Which field is in scope depends on which class defined the method
    • This is strange to me as a Ruby programmer.

Optional topic 2:
self/this => a function argument.
Is it contravariant?
No - it is treated 'covariant'.
