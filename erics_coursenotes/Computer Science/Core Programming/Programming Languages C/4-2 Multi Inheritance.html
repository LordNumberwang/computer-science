Multiple Inheritance

Traditional Multiple Inheritance would write something like this (note doesn't work in Ruby as written)

class C < A, B
    #...
end
class C < A, B
    #...
end

Instead in Ruby you copy code/behavior from one of the 'superclasses' and actually inherit from the other.

e.g.

class ColorPt3D_1 < ColorPt
  attr_accessor :z
  def distToOrigin
    Math.sqrt(x * x  + y * y + z * z)
  end
end

class ColorPt3D_2 < Pt3D
  attr_accessor :color
  def darken # error if @color not already set
    self.color = "dark " + self.color
  end
end

Mixins

Ruby unique.

A mixin is a collection of methods.
Less than a class. No instances of it.

Languages with mixins (e.g. Ruby modules) typically let a class have a superclass but include number of mixins.

Semantics: including a mixin makes its methods part of the class.
Extending or overriding in the order mixins are included in the class definition.
More powerful than helper methods because mixin methods can access methods and instsance variables on self not defined in the mixin.

Sample Mixin defined as a Ruby module:

module Doubler
  def double
    self + self # uses self's + message, not defined in Doubler
  end
end

class String
  include Doubler
end

module Doubler
  def double
    self + self # uses self's + message, not defined in Doubler
  end
end

class String
  include Doubler
end

Now rules on method look ups with mixins:

  • If looking for method m, look in object class first.
  • Then mixins (note: later mixins will shadow over earlier implementation of the method)
  • Then superclass
  • Then superclass' mixins

Instance Variables:

  • Mixin methods are included in the same object
  • BAD STYLE for mixin methods to use instance variables since name clash would be like CowboyArtist pocket problem.
  • Can run into collisions there

Two big mixins in Ruby:

  • Comparable - < > == != >= <= in terms of <=>
    • Define this 3 way comparison operator. Take 2 args, return negative (-1) if left is less, 0 if equal, + (1) if right is more.

Implementation for a Name class:

class Name
  attr_accessor :first, :middle, :last
  include Comparable
  def initialize(first,last,middle="")
    @first = first
    @last = last
    @middle = middle
  end
  def <=> other
    l = @last <=> other.last # <=> defined on strings
    return l if l != 0
    f = @first <=> other.first
    return f if f != 0
    @middle <=> other.middle
  end
end

  • Enumerable - define iterators (e.g. map, find, ... )
    • Define each and you get all the others for free

Each has to yield the appropriate object back to the block.

class MyRange
  include Enumerable
  def initialize(low,high)
    @low = low
    @high = high
  end
  def each
    i=@low
    while i <= @high
      yield i
      i=i+1
    end
  end
end

Interfaces

Java/C# examples.

Interfaces are about static typing (which Ruby does not have):

  • Preventing a method missing error

Each class you define introduces a type.
Declare methods - have argument type and result types.

class A {
    Object m1(Example e, String s) { ... }
    Integer m2(A foo, Boolean b, Integer i) {...}
}
class A {
    Object m1(Example e, String s) { ... }
    Integer m2(A foo, Boolean b, Integer i) {...}
}

If C is a transitive subclass of D, the C is a subtype of D.

  • Type-checking allows subtype anywhere supertype is allowed
  • Can pass Instance of C to a method expecting a 'D' instance.

Interfaces:

  • Also types, but not class - it is only a type.
  • Cannot put method definitions ... only their signatures (types)
interface Example {
    void m1(int x, int y) ;
    Object m2(Example x, String y) ;
}
interface Example {
    void m1(int x, int y) ;
    Object m2(Example x, String y) ;
}

Note the above has ';' no body is defined. You just declare methods.
Then pull in the implementation of Example within A:

class A implements Example {
    public void m1(int x, int y) {...}
    public Object m2(Example e, String s) {...}
}
class A implements Example {
    public void m1(int x, int y) {...}
    public Object m2(Example e, String s) {...}
}

Multiple Interfaces:

  • Interfaces provide no methods or fields (thus no duplication of method/fields), unlike multiple inheritance

Interfaces are for:

  • "Caller can give any instance of any class implementing Intf"
    • Callee can call methods in Intf, regardless of class
  • Much more flexible type system

Interfaces have little use in dynamically type language as a result.

Abstract Methods

What statically typed OOP language need to support required overriding
Required overriding

Often a class expects all subclasses to override some methods.
Ruby says: do not define must-override methods in superclass. Subclasses can add it.

def m1 v
    ... self.m2 e...
    end
def m2 v
    raise NotImplemented #should be defined in subclass
end
def m1 v
    ... self.m2 e...
    end
def m2 v
    raise NotImplemented #should be defined in subclass
end

Better solution in Java: use static checking to prevent this error
Use Java (abstract method) - give method signature, without definition
C#/C++ 'pure virtual methods'.

abstract class A {
    T1 m1(T2 x) { ... m2(e); ... }
    abstract T3 m2(T4 x); //abstract with no method body
}

class B extends A {
    T3 m2(T4 x) { ... } //which is defined in subclass instead
}

Passing code to other code...

Abstract methods and dynamic dispatch: An OOP way to give sub classes' code to superclass.

Higher order function: FP way to have caller 'pass code' to callee. Function as argument.
